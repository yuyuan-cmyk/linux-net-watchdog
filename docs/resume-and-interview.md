# 简历与面试说明

## 简历项目名称

基于 Linux 的网络设备状态监控工具

## 简历描述版本

**基于 Linux 的网络设备状态监控工具**  
技术栈：C、Linux、Socket、Makefile、日志模块

设计并实现一个运行于 Linux 环境的网络状态监控工具，模拟嵌入式设备网络诊断场景。项目支持通过配置文件管理检测目标，周期性检测目标 IP/域名的 TCP 端口可达性，记录检测结果、延迟和错误原因，并支持进程信号退出与日志持久化。

主要工作：

- 使用 C 语言与 POSIX Socket 实现 TCP 端口探测、非阻塞连接、超时控制和错误状态识别。
- 设计配置解析模块，支持按 `名称 主机 端口 超时时间` 加载多个检测目标。
- 实现日志模块，记录检测时间、目标地址、连接状态、延迟和失败原因，便于设备网络问题追踪。
- 使用 Makefile 管理编译流程，按配置、日志、监控、工具函数拆分项目结构。
- 面向嵌入式 Linux 网络诊断场景，验证端口关闭、目标不可达、连接超时等异常情况。

## 可以怎么讲

这个项目不是大型监控平台，而是一个偏底层的 Linux 网络诊断小工具。我主要想练习 C 语言在 Linux 下的 Socket 编程、配置解析、日志记录和 Makefile 构建。它可以放在网关、开发板或 Linux 设备上，周期性检查关键服务是否可达，比如网关、DNS、业务端口等。

## 面试常见问题

### 1. 为什么用 TCP 端口检测，不直接用 ping？

Ping 只能说明 ICMP 层面是否可达，但很多设备或服务器会禁用 ICMP。TCP 端口检测更接近业务服务状态，例如 80、443、22、53 等端口是否能建立连接。

### 2. 如何实现超时？

项目使用非阻塞 `connect`，如果返回 `EINPROGRESS`，再通过 `poll` 等待 socket 可写，并设置超时时间。最后用 `getsockopt(..., SO_ERROR, ...)` 判断连接是否真正成功。

### 3. 为什么要做日志模块？

嵌入式设备现场问题经常无法实时复现，日志可以记录故障发生时间、目标地址、失败原因和延迟，方便后续定位网络异常、服务不可达或链路不稳定。

### 4. 和嵌入式岗位有什么关系？

嵌入式软件并不只写驱动，也有大量 Linux 应用层、设备诊断、网络通信、日志和问题定位工作。这个项目对应的是嵌入式 Linux 设备的网络状态检测与故障分析场景。

### 5. 后续可以怎么扩展？

- 增加 ICMP 检测。
- 增加网卡状态读取，例如 `/sys/class/net`。
- 增加 JSON 或 INI 配置格式。
- 增加守护进程模式。
- 增加异常次数统计和自动重启网络服务。
- 移植到 ARM Linux 开发板。
